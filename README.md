# LLM Evaluation Benchmark - Football Knowledge

An end-to-end LLM research pipeline covering three phases: **baseline evaluation**, **RAG (Retrieval-Augmented Generation)**, and **DPO fine-tuning**. The project systematically measures what actually helps a language model answer football knowledge questions better - and what doesn't.

## Phase 1 - Baseline Evaluation

### Approach

Created a dataset of 70 football questions spanning four categories and three difficulty levels:

| Category | Description |
|----------|-------------|
| **CONCEPT** | Tactical and positional concepts (e.g. "What is a false nine?") |
| **HISTORY** | Facts, records, dates, and statistics |
| **COMPARE** | Comparing formations, styles, or players |
| **ANALYSIS** | Reasoning about why things happened |

Each question was run through two models served via the Groq API (chosen for its free tier and fast inference):

- **Llama 3.1 8B** (`llama-3.1-8b-instant`) - a compact, efficient model
- **Llama 3.3 70B** (`llama-3.3-70b-versatile`) - a much larger model with stronger factual recall

This produced 140 responses (70 questions × 2 models).

### LLM-as-judge scoring

Rather than scoring responses manually, **Claude Sonnet acted as an automated judge**. Each response was sent to Claude alongside a custom rubric that defined exactly what a high-quality answer must include. All categories are scored out of 10 points.

| Category | Rubric |
|----------|--------|
| **CONCEPT** | Accurate definition (3pts) + concrete example (3pts) + simple language (2pts) + when/why to use it (2pts) |
| **HISTORY** | Correct facts with exact numbers/dates (5pts) + relevant context (3pts) + no hallucinated stats (2pts) |
| **COMPARE** | Clear explanation of each side (3pts) + direct comparison (4pts) + when each is preferred (3pts) |
| **ANALYSIS** | Clear main argument (3pts) + 2 supporting reasons with real examples (4pts) + acknowledgment of complexity (3pts) |

Using a structured rubric per category (rather than a single generic prompt) makes the scoring consistent, auditable, and tuned to what actually matters for each question type.

### Baseline results

| Model | Overall | CONCEPT | HISTORY | COMPARE | ANALYSIS |
|-------|---------|---------|---------|---------|---------|
| Llama 3.3 70B | **7.44** | 7.50 | 8.33 | 7.50 | 6.40 |
| Llama 3.1 8B | 6.03 | 5.70 | 6.73 | 6.10 | 5.67 |

- The 70B model wins in every single category
- **Biggest gap: HISTORY (+1.6 pts)** - model size matters most for factual recall; larger models have simply seen more data during training
- **Smallest gap: ANALYSIS (+0.73 pts)** - both models struggle similarly with complex multi-step reasoning, suggesting reasoning ability scales less cleanly with size than factual memorization

## Phase 2 - Retrieval-Augmented Generation (RAG)

### Approach

RAG is a technique that retrieves relevant external information at query time and injects it into the prompt as context before the model generates its answer. The hypothesis: grounding models in Wikipedia content should improve factual accuracy, especially for HISTORY questions.

**Pipeline:**
1. Built a knowledge base by fetching Wikipedia articles on football topics
2. Split articles into chunks and encoded them as vectors using `sentence-transformers` (`all-MiniLM-L6-v2`)
3. Stored the vectors in a **FAISS** index (a library optimized for fast similarity search)
4. At query time, embedded each question and retrieved the most similar Wikipedia chunks
5. Injected the retrieved chunks as context into the prompt, then re-ran all 70 questions through both models
6. Re-evaluated all responses with the same Claude judge and rubric

### RAG results

| Model | Baseline | RAG | Change |
|-------|----------|-----|--------|
| Llama 3.3 70B | 7.44 | 6.60 | **−0.83** |
| Llama 3.1 8B | 6.03 | 5.76 | **−0.25** |

RAG hurt both models. The larger model was hurt more. Most affected category: **CONCEPT** (−0.83 pts).

### Why did RAG hurt?

This is a well-documented challenge in production RAG systems. Several failure modes compounded here:

- **Retrieval quality determines everything.** Wikipedia chunks were sometimes off-topic or only tangentially related to the question. When the retrieved context is noisy, it adds confusion rather than signal.
- **Larger models over-commit to retrieved context.** The 70B model, having stronger internal knowledge, actually suffered more because it actively tried to incorporate the retrieved chunks into its answer - even when those chunks were misleading. The 8B model, with weaker priors, partially ignored the context, which paradoxically helped.
- **Chunk size mismatch.** At 500 characters per chunk, many retrieved passages were too short to contain a complete answer, giving the model partial information rather than useful grounding.

The takeaway: RAG is not a free performance boost. Retrieval quality, chunk size, and the gap between the model's existing knowledge and the retrieved content all interact to determine whether RAG helps or hurts. Always evaluate empirically.

## Phase 3 - DPO Fine-Tuning

This is the most technically involved phase. The goal: fine-tune a small open-source model to produce better football answers, using preference data built from the baseline evaluation.

### What is DPO?

**DPO (Direct Preference Optimization)** is a fine-tuning technique that teaches a model to prefer better responses over worse ones. The traditional approach, RLHF with PPO (Proximal Policy Optimization), requires training a separate reward model and then using reinforcement learning to update the main model - a process that is complex, computationally expensive, and prone to instability.

DPO, introduced in a [2023 paper](https://arxiv.org/abs/2305.18290), achieves the same goal in a single training stage by directly optimizing the language model on preference pairs. For each training example, the model receives:

- A **prompt** (the question)
- A **chosen** response (the better answer)
- A **rejected** response (the worse answer)

Training adjusts the model's weights to increase the probability of generating the chosen response and decrease the probability of generating the rejected one. A temperature parameter (`beta=0.1`) acts as a regularizer, preventing the model from drifting too far from its original behavior.

### Why LoRA / QLoRA?

The base model, Llama 3.2 3B, has 3.2 billion parameters. Retraining all of them would require far more GPU memory than a free Colab instance provides.

**LoRA (Low-Rank Adaptation)** solves this by adding small trainable matrices to specific layers of the model, while the rest of the weights stay frozen. We trained only ~24 million parameters - **0.75% of the total** - while leaving the remaining 99.25% unchanged. Combined with **4-bit quantization (QLoRA)**, which compresses the frozen weights to use less memory, the entire training run fits on a free Google Colab T4 GPU (16GB VRAM).

### Step-by-step: how the fine-tuning was built

**Step 1 - Build the preference dataset**

`src/build_preferences.py` converts the baseline judge scores into preference pairs. For each question where one model scored **2 or more points higher** than the other, we create a pair: `chosen = higher-scoring response`, `rejected = lower-scoring response`.

The 2-point threshold is intentional. Smaller score differences may reflect judge variability rather than true quality differences, so we only include pairs where the gap is unambiguous. This produced **26 pairs**.

**Step 2 - First training run and diagnosis**

We trained Llama 3.2 3B on the 26 pairs. Results: **5.47 overall vs 6.03 baseline** - a regression. The diagnosis: 26 pairs is too few for a 3B model to learn stable preferences. With so little data, the model's fine-tuning updates disrupt its existing knowledge without providing enough signal to learn new behaviors reliably.

**Step 3 - Synthetic data generation via model distillation**

To fix the data scarcity problem, we used **model distillation** - using a larger, more capable model to generate training data for a smaller one.

`src/generate_synthetic_pairs.py` sent each of the 70 questions to Claude and asked it to generate two responses: a strong answer (chosen) and a deliberately weaker one (rejected). This yielded **69 additional pairs**, for a total of **95 pairs** in `preferences_merged.jsonl`.

This technique is powerful because it lets a small model benefit from the knowledge of a much larger one, without the compute cost of running the large model at inference time.

**Step 4 - Retrain with 95 pairs**

Training on the merged dataset produced dramatically better training metrics:

| Metric | v1 (26 pairs) | v2 (95 pairs) |
|--------|--------------|--------------|
| rewards/margin | 1.13 | **6.33** |
| Final loss | 0.423 | **0.200** |
| Accuracy | ~60–70% | **100% by step 10** |

The rewards/margin measures how confidently the model distinguishes chosen from rejected responses - a 5× increase is a strong signal that the model learned meaningful preferences this time.

**Step 5 - Serve locally with Ollama**

The fine-tuned adapter was merged back into the base model and converted to **GGUF format** (a quantized model format optimized for CPU/GPU inference). We used `q4_k_m` quantization, which produces a ~2GB file while preserving most of the model's quality. This GGUF file is then served locally using **Ollama** with a custom `Modelfile` that sets the system prompt and parameters.

Once set up, the model runs completely offline with no API costs.

## Final Results

All three model families evaluated by the same Claude judge with the same rubric:

| Model | Overall | CONCEPT | HISTORY | COMPARE | ANALYSIS |
|-------|---------|---------|---------|---------|---------|
| Llama 3.3 70B (baseline) | **7.44** | 7.50 | **8.33** | 7.50 | 6.40 |
| Llama 3.1 8B (baseline) | 6.03 | 5.70 | 6.73 | 6.10 | 5.67 |
| football-dpo-v2 (ours) | 5.27 | 5.35 | 4.67 | 5.30 | **5.73** |

### What the fine-tuned model got right

- **ANALYSIS** was the only category where our model nearly matched the 8B baseline (+0.07) - DPO successfully adjusted reasoning patterns and argument structure
- **Hard questions** showed the least regression (−0.16) - the model's behavior on complex, multi-step questions was least disrupted by training

### Where the fine-tuned model fell short

- **HISTORY** showed the largest regression (−2.07 vs 8B baseline) - this is the most important finding
- DPO can adjust *how* a model reasons and *how* it structures responses, but it cannot inject factual knowledge that isn't already in the base model's weights. Llama 3.2 3B simply knows fewer football facts than Llama 3.1 8B, and no amount of preference training can change that
- For factual-heavy domains, supervised fine-tuning on correct question-answer pairs would be more appropriate than DPO alone

## How to Run

### Setup

```bash
# 1. Clone the repo
git clone <repo-url>
cd llm-evaluation-benchmark

# 2. Create and activate a virtual environment
python -m venv venv
source venv/bin/activate       # Windows: venv\Scripts\activate

# 3. Install dependencies
pip install -r requirements.txt

# 4. Add API keys
echo "GROQ_API_KEY=your_groq_key" >> .env
echo "ANTHROPIC_API_KEY=your_anthropic_key" >> .env
```

### Phase 1 - Baseline

```bash
python src/run_models.py        # query both Groq models → data/responses.csv
python src/evaluate.py          # score with Claude judge → data/judge_scores.csv
```

### Phase 2 - RAG

```bash
python src/build_knowledge_base.py   # fetch Wikipedia, build FAISS index
python src/run_models_rag.py         # query models with RAG context → data/responses_rag.csv
python src/evaluate.py \
  --input data/responses_rag.csv \
  --output data/judge_scores_rag.csv
```

### Phase 3 - DPO Fine-Tuning

```bash
# Build preference pairs from judge scores
python src/build_preferences.py          # → data/preferences.jsonl (26 pairs)

# Generate synthetic pairs via Claude
python src/generate_synthetic_pairs.py   # → data/synthetic_pairs.jsonl (69 pairs)
# (preferences_merged.jsonl is created automatically - 95 total pairs)

# Train: open in Google Colab with a T4 GPU (free tier)
# notebooks/train_dpo.ipynb
# After training, export the GGUF file and place it where your Modelfile references it

# Serve locally with Ollama
ollama create football-dpo-v2 -f models/Modelfile

# Evaluate the fine-tuned model
python src/run_finetuned.py
python src/evaluate.py \
  --input data/responses_finetuned_v2.csv \
  --output data/judge_scores_finetuned_v2.csv

# Compare all models
python src/compare_results.py
```

### Analysis and Demo

```bash
# Full analysis notebook with charts
jupyter notebook notebooks/analysis.ipynb

# Interactive Streamlit demo
streamlit run src/app.py
```

## Development Approach

This project was built with Claude Code as a development assistant - used for generating boilerplate code and speeding up implementation of well-understood components.

All architectural decisions were made independently: which models to compare, how to design the rubric, the choice of DPO over SFT, the 2-point preference threshold, diagnosing why the first training run regressed, and the decision to use synthetic data generation to fix the data scarcity problem.

## Key Takeaways

**1. Model size matters most for factual knowledge, least for reasoning.**
The 70B and 8B models were closest on ANALYSIS questions (+0.73 gap) and furthest apart on HISTORY (+1.6 gap). For complex reasoning tasks, a smaller model is far more cost-competitive than you might expect.

**2. Naive RAG is not always better than no RAG.**
Both models performed worse with RAG than without it. The larger model was hurt more because it tried harder to use the noisy retrieved context. Retrieval quality, chunk size, and the model's existing knowledge all interact - always run an ablation before committing to RAG in production.

**3. DPO can improve reasoning style but cannot inject factual knowledge.**
The fine-tuned 3B model improved on ANALYSIS questions but regressed significantly on HISTORY. DPO adjusts *how* a model reasons; it cannot compensate for facts that aren't in the base model's weights. For factual domains, supervised fine-tuning on correct answer pairs is a better starting point.

**4. Data quality beats data quantity for preference training.**
26 pairs was too few and caused regression. 95 pairs (including 69 Claude-generated synthetic pairs) produced stable, meaningful training signals. Model distillation - using a large model to generate training data for a smaller one - is a practical way to bootstrap a preference dataset when real preference data is scarce.

**5. The full pipeline is end-to-end reproducible at low cost.**
Baseline evaluation and synthetic data generation use API calls (Groq free tier + Anthropic). Fine-tuning runs on a free Colab T4. Inference runs locally via Ollama with no ongoing costs. The entire research cycle from questions to a served fine-tuned model requires only API credits.
